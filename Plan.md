# 기업 중복수혜 검색 시스템 기획서 및 구현 계획 (지원 이력 관리 추가 반영안)

본 문서는 내부 기업 정보를 관리하고 외부 엑셀 데이터를 업로드하여 중복수혜 여부를 검증하는 시스템의 확정된 기획 및 구현 계획입니다. 추가로 요청하신 **'과거 지원 이력 관리 기능'** 및 **'포기/제외 체크 항목'**이 반영되어 있습니다.

## 1. 시스템 개요 (Goal Description)
* **목적**: 외부에서 접수되는 기업 목록(엑셀)을 내부 시스템에 저장된 기존 기업 데이터와 대조하여, 중복 수혜 기업을 빠르고 정확하게 찾아내는 시스템 구축.
* **주요 기능**:
  1. **내부 데이터 관리**: 기업 정보 단건 및 엑셀(XLSX) 대량 등록, 수정, 삭제, 기업 DB 관리 대시보드 제공
  2. **과거 지원 이력 관리 (NEW)**: 각 기업이 과거에 어떤 사업으로 얼마의 지원을 받았는지 개별 이력 관리 및 상태(선정/완료/포기/제외) 관리
  3. **중복 조회 (엑셀 업로드)**: 엑셀 파일을 업로드(요청기관, 문서번호, 요청내용 포함)하여 내부 데이터와 대조
   4. **검색 로직 (개편)**: 
     * **사업자등록번호 기반 일치 판별**:
       * **Exact Match (1순위)**: 입력받은 사업자등록번호가 내부 DB의 사업자번호와 정확히 일치하면 무조건 대조 및 확인 필요(EXACT) 대상으로 분류합니다.
       * **Typo Match (2순위)**: 일치하는 번호가 없더라도, 사업자번호의 편집 거리(Levenshtein Distance)가 2 이하로 매우 유사하고 **동시에** 기업명 유사도가 높거나 소재지(시/군)가 일치하는 경우 오타로 판단하여 유사 매칭(FUZZY) 후보로 노출합니다.
       * **신규 기업 판정**: 위 두 가지 조건에 해당하지 않으면 별도의 유사도 검색 없이 즉시 안전한 신규 등록 기업(NEW)으로 판단하여 불필요한 대조를 방지합니다.
     * **이력 필드 키워드 기반 중복 의심 판별 (우선순위 적용)**:
       * 매칭된 기업의 신청 지원분야(`supportField`)와 과거 지원 이력(상태가 '선정' 또는 '완료')의 각 필드를 다음 **우선순위 순서**로 GBSA 대표 키워드 오버랩 여부를 검사합니다:
         1. **지원과제명** (`project_name`) — 가장 구체적인 과제 정보로 1순위 검색
         2. **지원사업명** (`program_name`) — 사업 카테고리 수준으로 2순위 검색
         3. **비고** (`notes`) — 보조 텍스트로 3순위 검색
       * 가장 먼저 키워드가 매칭되는 필드에서 결과를 반환하며, 매칭 시 **[중복 수혜 의심]** 플래그 및 안내 메시지를 표시합니다.
  5. **결과 리포트**: 매칭된 결과를 화면에 표시하며, 매칭된 기업의 **과거 지원 내역과 총 누적 지원 금액**을 함께 표시 및 엑셀 다운로드 지원
  6. **인증 시스템**: 시스템 접근을 제한하기 위한 간단한 로그인 기능 구현

## 2. 데이터 구조 (Data Schema)
기업 기본 정보와 과거 지원 이력을 분리하여 **1:N (일대다) 관계**의 두 가지 테이블로 나누어 관리합니다. 외부 데이터베이스를 사용하므로 개인정보(대표자명 등)는 배제합니다.

### A. 기업 기본 정보 테이블 (Companies)
* `id` (UUID) - 시스템 내부 고유키
* `사업자등록번호` (Business Registration Number) - 하이픈(-) 제외 등 정규화 저장 (예비 창업자는 NULL 허용)
* `업체명` (Company Name)
* `주소` (Location) - 시/군명을 포함한 상세 주소 전체 저장 (예: "경기도 수원시 영통구 광교로 156")
* `주요제품` (Main Products)
* `지원분야` (Support Field)
* `추가정보` (Additional Data / JSONB) - 향후 새로운 항목 추가 대비용

### B. 과거 지원 이력 테이블 (Support_Histories)
하나의 기업이 여러 번 지원받을 수 있으므로 별도 테이블로 구성합니다.
* `id` (UUID) - 이력 고유키
* `company_id` - 기업을 식별하기 위한 외래키 (Foreign Key)
* `business_number` (varchar) - **[자동 동기화]** `company_id`에 연결된 `companies.business_number` 값이 DB 트리거에 의해 자동으로 채워짐 (사용자 직접 입력 불가)
* `지원사업명` (Program Name / `program_name`) - 과거에 지원받은 사업 카테고리 이름
* `지원과제명` (Project Name / `project_name`) - **[NEW]** 지원받은 세부 과제명 (선택 입력). **중복 키워드 검색 1순위 대상**
* `상태` (Status) - 선정, 완료, 포기, 제외 (UI에서 셀렉트박스로 관리)
* `선정금액` (Selected Amount) - 사업 선정 당시의 1차 금액
* `지원금액` (Support Amount) - 최종 지원받은 금액
* `년도` (Year) - 지원 연도
* `비고` (Notes) - 기타 특이사항 (포기/제외 사유 등). **중복 키워드 검색 3순위 대상**

## 3. 기술 스택 및 배포 방식 (Tech 스택 & Deployment)
GitHub Pages와 Vercel 환경을 모두 지원할 수 있는 하이브리드 아키텍처를 구성합니다.

* **Frontend Framework**: **Next.js** (React 기반, Turbopack 최적화 빌드 완료)
* **Styling**: **Tailwind CSS v4** (GBSA 브랜드 정체성이 녹아든 HSL 기반 테마 및 엔터프라이즈 레이아웃 디자인)
* **Database & Auth**: **Supabase** (PostgreSQL 기반 클라우드 DB) 및 **Local Storage** 하이브리드 연동 완료
  * `isSupabaseConfigured` 상태 체크를 통해 Supabase 설정 여부에 따라 원격 PostgreSQL DB 또는 로컬 `localStorage`를 자동 선택하여 데이터를 영구 보존(Persistence)합니다.
  * 데이터 매핑 레이어(Mappers)를 도입해 PostgreSQL의 `snake_case` 컬럼 스키마와 프론트엔드의 `camelCase` 데이터 타입을 양방향으로 매핑하며, JSONB(`additional_data`)를 활용해 매칭 메타데이터를 보존합니다.
  * DB 작업의 비동기성(Async/Await)을 프론트엔드 서비스 레이어(`companyService`) 및 컴포넌트 전체에 완벽히 적용하였습니다.
* **주요 라이브러리 및 유틸리티**: 
  * `exceljs` (클라이언트 측 엑셀 업로드 파일 파싱 및 GBSA 테마 스타일이 적용된 다운로드용 결과 엑셀 리포트 생성)
  * `fuse.js` (브라우저 내 실시간 유사도 검색 및 정확도 백분율 매칭 엔진)
  * `utils/format` (사업자등록번호 하이픈 자동 포맷팅 `formatBusinessNumber`, 전체 주소 기반 시/군 추출 `extractSiGun` 기능 제공)

## 6. 소재지 및 주소 데이터 관리 규칙 (Data Management Rules) - [NEW]
* **기본 정의**:
  * **주소 (Location / Address)**: 데이터베이스의 `companies` 테이블 내 `location` 컬럼에는 항상 상세 주소(전체 도로명 주소 등)가 저장되어야 합니다.
  * **소재지 (City/County)**: 화면상의 지역별 분류 필터링 및 조회를 돕기 위한 요소로, 데이터베이스에 별도 컬럼으로 저장하지 않습니다. 대신 전체 주소(`location`)에서 시/군 단위명(예: 수원시, 성남시)을 프론트엔드가 `extractSiGun` 헬퍼 함수를 통해 실시간으로 파싱하여 사용합니다.
* **프론트엔드 컴포넌트 구현 시 필수 수정 사항 (버그 수정 가이드)**:
  * **`DBManageModal.tsx` (기업 정보 수정 모달)**:
    * 기존 버그: 수정 폼 제출 시 DB의 `location` 컬럼에 세부 주소(`editAddress`)가 아닌 시/군명(`editLocation`)이 전달되어 상세 주소가 유실되는 현상이 있었습니다.
    * 조치 사항: `companyService.updateCompany` 호출 시 `location: editAddress`가 입력되도록 바인딩을 전면 교체해야 합니다.
  * **`NewCompanyModal.tsx` (신규 기업 등록 모달)**:
    * 기존 버그: 단건 신규 기업 등록 시 DB의 `location` 컬럼에 세부 주소(`address`)가 아닌 시/군명(`location`) 상태값이 매핑되어 등록되는 오류가 있었습니다.
    * 조치 사항: 등록 객체 생성 시 `location: address`가 입력되도록 수정해야 합니다.
